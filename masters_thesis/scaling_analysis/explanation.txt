====================================================================
SCALING ANALYSIS — WHAT EACH CALCULATION DOES AND WHY
====================================================================
This document explains, calculation by calculation, the analysis in
`analyze.py`. The numbers below (1-6) match the CALCULATION blocks in the
script exactly. Run `python3 analyze.py` to reproduce all figures.

THE DATA (data/scaling_records.csv)
  One row = one RUN of the shared-git scaling experiment: a fixed pool of K
  mutually-conflicting features is split across N agents, who each implement
  their share, then merge their peers' work on a shared repo and rebuild one
  integrated patch. We then run every feature's held-out test suite against
  that single integrated tree.
  Columns used:
    pool_id    distinct clique = repo/task/features (a task can appear as >1 clique)
    N          number of agents the K features were split across (1..4)
    score      graded correctness = n_passed / K  (fraction of feature suites passing)
    all_passed True iff every feature passed (n_passed == K)
    cost       API list-price USD for the run, summed over the N agents
  Totals: 148 runs, 14 distinct clique-pools. N=1,2,3 have all 14 pools;
  N=4 has only the 6 pools whose workload was K=4 (so 4 agents fit).


--------------------------------------------------------------------
1. AGGREGATE BY N  (mean_score, all_pass_rate, mean_cost)
--------------------------------------------------------------------
WHAT: Group all runs by agent count N and take, per N, the average graded
      score, the fraction of runs that were FULLY correct (all_pass rate), and
      the average cost.
WHY : These are the raw "as you add agents" curves. They establish the two
      basic facts the whole study rests on:
        - correctness falls  (mean_score 0.96 -> 0.89 over N=1..3;
          all_pass rate 0.89 -> 0.82 -> 0.77 -> 0.69 monotonically),
        - cost rises steeply ($0.78 -> $7.21).
      The all_pass RATE is the strict, honest success measure (did the team
      ship a fully-working result); mean_score is the finer graded version
      that credits partial success (3 of 4 features = 0.75).
NOTE: The N=4 mean_score ticks up to 0.92 only because N=4 contains just the 6
      cleaner K=4 pools, not all 14 — a composition change, NOT a recovery.
      This is precisely why we lean on efficiency (calc 2-3), which is immune
      to it: those N=4 pools still cost ~$7, so their efficiency is the floor.


--------------------------------------------------------------------
2. EFFICIENCY  (work solved per dollar, by N)
--------------------------------------------------------------------
WHAT: efficiency(N) = mean_score(N) / mean_cost(N). Also expressed as a
      percentage of the solo (N=1) value.
WHY : This is the metric that combines both effects into one honest number:
      you are paying more (cost up) for the same-or-worse output (score flat/
      down), so the useful work you get per dollar drops sharply — to 35%,
      19%, and finally 10% of the solo value at four agents. It answers the
      practical question directly: "is adding agents worth it?" (No.)
      We use the ratio-of-means (mean_score/mean_cost) as the headline; the
      mean-of-per-run-ratios gives the same shape with larger absolute numbers.


--------------------------------------------------------------------
3. POWER-LAW FIT of efficiency vs N   (THE headline result)
--------------------------------------------------------------------
WHAT: Fit efficiency = a * N^(-b) to the four aggregate efficiency points.
      Done as ordinary least squares on the log-log transform
      ln(efficiency) = ln(a) - b*ln(N), then report a, b, and R^2.
      Result: a = 1.28, b = 1.61, R^2 = 0.996.
WHY : A single, tight law (R^2 ~ 1) is far more compelling than a vague
      "it goes down": it says efficiency falls by a FIXED multiplicative factor
      every time you double the team — 2^(-1.61) ~ 0.33, i.e. each doubling of
      agents leaves you ~one-third the work per dollar.
      The exponent b is the substantive claim:
        - a perfect split (each agent does 1/N of the work at fixed cost) would
          give b = 1 (efficiency ~ 1/N);
        - we measure b = 1.61 > 1, meaning the loss is SUPER-PROPORTIONAL —
          worse than simple division. The gap above 1 is the pure coordination
          overhead: redundant context each agent loads, messaging, re-integration,
          and repairing botched merges.


--------------------------------------------------------------------
4. PER-POOL POWER LAW  (universality check)
--------------------------------------------------------------------
WHAT: Repeat the calc-3 fit separately for each clique-pool (pools with >=3
      N-levels), and report the range/mean of the exponents b and whether every
      fit is tight.
      Result: all 14 pools fit; b in [1.12, 2.25], mean ~1.72, every R^2 > 0.9.
WHY : This proves the power-law collapse is not an artifact of averaging or of
      a few bad pools — it holds for EVERY task, including ones that stay 100%
      correct at all N. So the efficiency penalty is a property of dividing the
      work at all, not of any particular task failing. It makes the result
      robust and general rather than cherry-picked.


--------------------------------------------------------------------
5. DEGRADER CLASSIFICATION  (correctness is task-dependent)
--------------------------------------------------------------------
WHAT: For each pool, spread = max_N(mean_score) - min_N(mean_score); flag the
      pool as "degrades" if spread >= 0.12 (about half a feature out of K=3/4).
      Result: 6 of 14 pools degrade.
WHY : Separates the two distinct effects and keeps us honest. Correctness loss
      is NOT universal (unlike the efficiency collapse) — it concentrates on the
      hard, conflict-dense tasks: all four dspy cliques degrade (dspy is THE
      degrader repo), plus llama/17244 at K=3 and tiktoken's stochastic K=4 dip.
      Eight pools integrate cleanly. This tells the reader where the *correctness*
      risk of adding agents lives (hard tasks), as opposed to the *efficiency*
      penalty, which is everywhere.


--------------------------------------------------------------------
6. COST vs N  (near-linear per agent)
--------------------------------------------------------------------
WHAT: For pools present at both N=1 and N=4, the per-agent cost increment
      (cost(4)-cost(1))/3, plus the overall N=4/N=1 cost ratio.
      Result: each added agent costs a roughly constant ~$2 (range ~$1.2-3.1);
      total cost ~9-10x from one to four agents.
WHY : Explains the mechanism behind the power law. Cost grows about LINEARLY in
      N (each agent adds a fixed cost: its own context load plus its slice of
      the work). Linear cost divided into flat-or-falling correctness is exactly
      what produces the super-linear efficiency collapse in calc 3. So calc 6 is
      the "why" underneath the headline number.


--------------------------------------------------------------------
ONE-LINE CONCLUSION (what all six add up to)
--------------------------------------------------------------------
Splitting a task a single agent can already solve across more agents delivers
no more correctness (and less, on hard tasks) while costing ~linearly more per
agent — so the work delivered per dollar falls as a power law, ~N^-1.6, roughly
one-third as efficient each time you double the team, on every task tested.
