================================================================================
NANO STUDY — DOES INTER-AGENT MESSAGING IMPROVE MULTI-AGENT CODING?
Coop (no messaging) vs Coop (messaging), on the nano coordination benchmark
Date: 2026-07-02  (updated 2026-07-04 to k=15)   Model: claude-sonnet-5 (claude_code wrapper)
================================================================================

--------------------------------------------------------------------------------
DATASET: WHY WE BUILT IT & HOW
--------------------------------------------------------------------------------
Why not an off-the-shelf subset:
  To measure whether a COMMUNICATION protocol helps, the benchmark must isolate
  COORDINATION as the variable. Existing subsets (flash/lite) mix in two kinds of
  pair that hide any protocol effect:
    (a) capability-floored pairs -- the model cannot implement the features at
        all, so they fail in every condition (0 with messaging, 0 without). They
        add cost and noise but no signal.
    (b) no-conflict pairs -- the two features do not really overlap, so the naive
        merge already succeeds. There is nothing to coordinate, so a protocol
        cannot move them either.
  Averaged over a set full of (a) and (b), a real coordination effect gets
  diluted toward zero. We needed a set where EVERY pair is "coordination-limited":
  the work is doable, but only if the agents coordinate.

Definition of a coordination-limited pair:
    (1) each feature is individually implementable by the model (capability
        present), AND
    (2) the two features gold-conflict, so the naive un-coordinated merge breaks.
  Then any failure is attributable to coordination, and a protocol has room to
  help -- the pair is a sensitive instrument, not dead weight.

How we built it (all reproducible in scripts/nano/):
  1. Frame: all gold merge-conflict pairs, restricted to Python -- single
     language removes the language x repo confound and is the only language with
     enough tasks to give many independent clusters.
  2. Capability screen: ran candidate pairs SOLO (one agent, both features, no
     merge) on the target model. Solo success shows the features are buildable;
     solo is also the right capability signal (the no-messaging control cannot be
     used -- a merge conflict there fails the tests regardless of capability).
     ~99 solo runs across four screening rounds.
  3. Select: keep pairs whose features are individually solvable; exclude
     capability-floored tasks (no feature solvable even solo). One pair per
     distinct task -> independent clusters for valid clustered statistics.
  4. Result: 20 coordination-limited pairs, one per task, across 9 Python repos.

Payoff: because capability and "does the conflict bite" are controlled for by
construction, a measured control-vs-messaging difference is attributable to the
protocol -- which is exactly what let a significant effect emerge at only k=5.

--------------------------------------------------------------------------------
METHOD
--------------------------------------------------------------------------------
Dataset (dataset/subsets/nano.json):
  20 gold merge-conflict pairs, one per distinct Python task, across 9 repos.
  Every pair is "coordination-limited": a solo screen (99 solo runs) confirmed
  each feature is individually implementable by the model, so the two features
  CAN be built -- coordination, not raw capability, is the variable under test.
  Capability-floored tasks (no feature solvable solo) were excluded.

Design:
  Two conditions on the same 20 pairs:
    - control  : coop, --no-messaging  (agents cannot talk)
    - protocol : coop, messaging on    (agents have send_message)
  k = 15 replicate runs per pair per condition = 600 pair-runs total,
  all evaluated (patches merged + tests run per feature).  (Originally
  reported at k=5; extended to k=15 on 2026-07-04. The extra data
  attenuated the effect substantially -- see FINDING.)
  Concurrency 2 (higher oversubscribes the 12-core host during build/test bursts
  and causes agent-startup failures -- established by smoke tests).

Analysis (scripts/nano/analyze.py):
  Per-pair pass-rates with Wilson CIs; pooled effect via Cochran-Mantel-Haenszel
  stratified by task (respects the k-replicate clustering -- no pseudoreplication).
  2 "ceiling" pairs auto-excluded (the naive no-messaging merge already passes
  them >60% of the time -> no conflict to coordinate), leaving 18 pairs.
  Primary endpoint  : merge-clean rate (coordination-specific).
  Secondary endpoint: both_passed (full end-to-end task success).

--------------------------------------------------------------------------------
FINDING
--------------------------------------------------------------------------------
  ENDPOINT              CONTROL   MESSAGING   EFFECT (CMH)    p-value
  merge-clean (primary)   13%       21%       OR = 1.79       0.021    *
  both_passed (secondary)  2%        3%       OR = 1.90       0.52     n.s.
  (18 pairs, 270 runs per arm after ceiling exclusion)

  >> Messaging still significantly raises the clean-merge rate (13% -> 21%,
     p ~ 0.02) -- a real coordination win.
  >> It does NOT improve end-to-end task success (both_passed flat, not sig.).

  NOTE on the k=5 -> k=15 change: the ORIGINAL writeup (k=5) reported
  merge-clean 22% -> 43% (OR 2.73, p=0.006). Tripling the data more than halved
  the effect (OR 1.79) and moved it from robustly to marginally significant
  (p=0.02, near the 0.05 line). The k=5 sample was favourable; this is textbook
  regression to the mean. Treat the direction as solid and the magnitude as
  modest -- messaging helps merges by a handful of points, not 2x.

Interpretation:
  Agents that can message each other genuinely conflict less -- they edit
  less-overlapping code / negotiate boundaries, so their patches merge cleanly
  somewhat more often. But a clean merge is not sufficient here: the features still
  fail their tests independent of the merge, so full success does not rise.
  This is a clean dissociation: the protocol fixes COORDINATION but not the
  remaining CAPABILITY/correctness bottleneck.

Supporting detail:
  hf/3997 [1,5] -- the pair specifically added because it is coordination-limited
  (each feature doable alone, but not both by one solo agent) -- went
  0% -> 20% both_passed with messaging (0/15 -> 3/15), the single largest
  task-success gain, and the ONLY pair where messaging unlocked any end-to-end
  success. Its per-feature passes move in lockstep (feat1 0->3, feat2 0->3, both
  0->3): when the features ARE reliably buildable, messaging converts a
  coordination gain into joint success. (At k=5 this pair read 0% -> 40%, i.e.
  2/5; the true rate is nearer 20%.)

--------------------------------------------------------------------------------
WHY DO THE TASKS FAIL? (k=15 failure-mode breakdown)
--------------------------------------------------------------------------------
The near-zero both_passed rate is NOT mainly a merge problem. Reading the
per-feature eval fields (feature1.passed / feature2.passed) across all 600
pair-runs shows the merge is largely a red herring for end-to-end success:

  - For 14 of the 18 analysed pairs, EACH feature passes ~0/15 in BOTH arms --
    even on the runs where the merge is textually CLEAN. A clean merge of two
    non-working halves still fails. e.g. dspy/8635 [1,3] merges clean 6/15 in the
    messaging arm, but feature2 passes 0/15 across every run, clean or not.

  - So the binding constraint is per-feature test-passing in the coop split, not
    conflict resolution. Messaging moves merge-clean (13->21%) but that gain has
    nothing to convert, because the underlying feature implementations don't pass.

  - The pairs that DO pass end-to-end are exactly the ones where the features are
    reliably buildable: the two excluded ceiling pairs (llama_index/18813,
    pallets_click/2956; feat1/feat2 ~11-14 of 15), pillow/25 (~5/15 both arms),
    and hf/3997 (0->3 with messaging). Everywhere the features build, both_passed
    is non-zero; everywhere they don't, no protocol can help.

Leading hypothesis (to be checked against the solo screen -- see NEXT STEPS):
  The capability screen admitted a pair if each feature EVER passed solo (>=1 of
  N solo runs). That is a weak bar: a feature that passes solo ~1/5 of the time
  clears it but is near the capability floor. Getting BOTH such features to pass
  in one merged submission is then ~ p1*p2 ~ a few percent -- which is exactly
  the both_passed rate we observe. If so, the set is capability-floored, not
  coordination-limited, for most pairs, and the right fix is to re-screen on solo
  PASS-RATE (e.g. keep features that pass solo >=60%), not mere solo feasibility.

Implication for the protocol/message-schema work:
  A richer message schema can only help pairs that are coordination-limited
  (buildable but conflict-prone). Today only ~1-2 pairs (hf/3997, arguably
  pillow/25) meet that bar; the rest are floored on raw capability. Before
  investing in schema experiments, re-curate the set to features with high solo
  pass-rates so there is headroom for coordination to matter.

--------------------------------------------------------------------------------
CAVEATS
--------------------------------------------------------------------------------
  - k = 15: the merge-clean effect is significant but modest (p=0.02); both_passed
    is near-floor and underpowered; a small true effect there could be missed.
  - Inference is conditional on these 20 Python pairs and claude-sonnet-5;
    single-language by design (not a cross-language generalization claim).
  - "merge-clean" counts a clean textual merge; it does not require the merged
    code to be correct -- it is a coordination proxy, not a success measure.

--------------------------------------------------------------------------------
NEXT STEPS
--------------------------------------------------------------------------------
  1. RE-CURATE THE SET ON SOLO PASS-RATE (highest priority). The current screen
     admitted a pair if each feature ever passed solo; the k=15 failure breakdown
     shows most pairs are actually capability-floored (feat1/feat2 ~0/15 in the
     coop split). Re-screen keeping only features with a high solo pass-rate
     (e.g. >=60% of solo runs), so there is real headroom for coordination -- and
     a message-schema experiment -- to move both_passed. Without this, any
     protocol study is measured against a near-floor and cannot show an effect.
  2. Test a richer MESSAGE SCHEMA (the protocol-mode framework): free vs typed
     vs field-structured (e.g. an `ownership` schema forcing agents to declare
     which files they own). Only meaningful AFTER step 1 -- run it on the
     re-curated, coordination-limited set.
  3. Deeper read of the surviving merge-clean-but-fail runs: distinguish (a) the
     feature was never implemented in the coop split, (b) the merge dropped it,
     (c) implemented-but-wrong (capability). Sample transcripts + test_output.
     (k=15 aggregate already points at capability; this confirms the mechanism.)
